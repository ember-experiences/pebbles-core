## What the CLI tells us (our north star)

`cli/main.py` imports:
- `from pebbles.engine import Engine` → the **first variant** in engine.py (protocol-based, lines 1-134)
- `from pebbles.storage import Storage` → the **second variant** in storage.py (JSON-based, lines 146-end)
- `from pebbles.sources.hackernews import HackerNewsSource`
- `from pebbles.delivery.telegram import TelegramDelivery`

The CLI defines inline: `SimpleKeywordMatcher`, `NoOpFilter`. So the intended shipping shape is:

```
Engine (protocol-based)
  ├─ Source protocol
  ├─ Matcher protocol  ← SimpleKeywordMatcher satisfies
  ├─ Filter protocol   ← NoOpFilter satisfies
  ├─ Delivery protocol ← TelegramDelivery satisfies
  └─ Storage           ← JSON-file Storage

Pebble (model) — a content item
  — uses 'description' + 'metadata' fields (from first variant of models.py)
```

This is the simplest, most protocol-clean version. The other variants (SQLite storage with rich delivery tracking, `PebbleEngine` with `Recipient`/`Interest` matching, Twilio delivery) are more ambitious but weren't wired into the CLI.

## Decisions I need from you

### Decision 1 — which `Engine` variant is canonical?

Four variants in engine.py:

| Variant | Lines | Shape | CLI uses it? |
|---|---|---|---|
| **A. `Engine` (protocol-based)** | 1-134 | `Engine(sources, matcher, filter, delivery, recipient, storage)`. Simple, orthogonal protocols. Deduplicates by URL. | ✅ yes |
| B. `PebbleEngine` (Twilio) | 135-240 | Config-driven. Round-robin to SMS recipients. | ❌ |
| C. `PebbleEngine` (collector) | 241-340 | Separates `collect()` from `get_unsent_pebbles()`. Store-then-fetch flow. | ❌ |
| D. `PebbleEngine` (matcher-aware) | 341-620 | Uses `InterestMatcher` + `Recipient` models for priority-scored delivery. Richest version. | ❌ |

**My recommendation: keep A, delete B + C + D.** Reason: it's what the CLI expects, it's the cleanest protocol-based shape, and it matches our Pebbles family plan where `pebbles-core` extends with `Principal` / `Queue` / `ApprovalChannel` primitives for Scout + Presence. Adding `Recipient`/`Interest` scoring (from D) can come back as an optional helper if Pebbles Family needs it later.

**Alternative I'm willing to consider:** keep A + keep D's `InterestMatcher` as a separate module. D is the most feature-complete matcher. If you want "the matching the README describes" (semantic scoring, daily limits, priority ordering) preserved for Family-tier work, this is the one.

**Your call:** A only, or A + D's matcher preserved?

### Decision 2 — which `Pebble` / `Interest` / `Recipient` model is canonical?

Two variants in models.py:

| Variant | Pebble fields | Recipient fields | Interest | Used by |
|---|---|---|---|---|
| **A. First variant** | title, url, **description**, source, timestamp, metadata | name, interests, **delivery_method + delivery_address**, max_daily_pebbles, metadata | Has `matches()` keyword method | Matches CLI + first engine variant |
| B. Second variant | title, url, **content**, source, timestamp | name, **phone**, interests, max_daily_pebbles | No matches() method | Older draft (Recipient.phone = Twilio era) |

**My recommendation: keep A.** Reason: it matches what CLI uses (`pebble.description`, `pebble.metadata`), it has the multi-delivery-channel shape (`delivery_method` can be telegram/email/webhook) which is where Pebbles Family needs to go, and `Interest.matches()` is useful.

Note: `__init__.py` also tries to import `SourceType` + `DeliveryMethod`. **Neither exists.** I propose we either (a) delete those imports since they're not needed, or (b) add them as minimal enums if we think they document the supported options well. Recommend (a) for v0.1 release, (b) later if useful.

**Your call:** A + drop the ghost imports? Or do you want me to introduce `SourceType`/`DeliveryMethod` enums?

### Decision 3 — which `Storage` variant is canonical?

Two variants in storage.py:

| Variant | Backend | Tracks | Used by |
|---|---|---|---|
| A. `PebbleStorage` (SQLite) | SQLite | pebbles + deliveries with scores | nothing currently |
| **B. `Storage` (JSON)** | JSON file | just `(url, recipient, timestamp)` triples | CLI imports this |

**My recommendation: keep B for v0.1 ship, build A back as `SupabaseStorage` later for Phase A of our family plan.** Reason: B is what the CLI uses, it's the simplest "does it work" version, and our Pebbles family plan already has Supabase storage as a Phase A deliverable — so the richer SQLite A is effectively a bridge we don't need (we'll skip straight to Supabase).

**Note:** CLI's `status` command has a bug — references `storage.conn` which only exists on A. Either we fix `status` to use B's API, or we drop the `status` command from the v0.1 ship. I recommend fixing it (trivial — use B's `get_stats()` which already returns what status needs).

**Your call:** B only + fix status command?
